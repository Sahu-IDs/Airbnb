Slide 02 · Problem Statement

Why This Problem Exists?

😣

No Unified Property Platform

Landlords and tenants in smaller cities have no centralized digital platform to connect — relying on physical notices or middlemen.

💥

"Works on My Machine" Problem

Applications run fine on the developer's laptop but crash in production due to different Node.js versions, missing packages, or OS differences.

📉

No Scalability

Traditional single-server deployments crash under heavy user traffic. There is no automatic way to add more servers when needed.

🔒

Insecure Config Management

API keys and passwords are hardcoded in source code — leaked in Git repositories, causing security breaches.

Slide 03 · Objectives

Project Objectives

01

Build a Full-Stack CRUD Web App

Develop a functional Airbnb clone with Create, Read, Update, Delete operations for property listings — backed by MongoDB Atlas.

02

Containerize using Docker

Package the entire Node.js application into a portable, lightweight Docker image using a multi-stage Dockerfile so it runs identically on any machine.

03

Orchestrate with Kubernetes

Deploy the container on a Kubernetes cluster with multiple replicas, auto-scaling (HPA), rolling updates, and self-healing capabilities.

04

Implement Secure Secret Management

Inject environment variables (API keys, DB URL) securely through Kubernetes Secrets — never hardcoded in source code or Docker image.

05

Achieve Zero-Downtime Deployment

Enable seamless application updates using Kubernetes Rolling Update strategy — old pods remain active until new ones are fully healthy.

Slide 09 · Kubernetes Manifests

K8s Configuration Files

📁 namespace.yaml

kind: Namespace
metadata:
  name: airbnb

Isolates all app resources in a dedicated namespace.

🔐 secret.yaml

kind: Secret
type: Opaque
data:
  ATLASDB_URL: ***base64***
  CLOUD_API_KEY: ***base64***

Secrets stored as Base64. Never in source code.

🚀 deployment.yaml

replicas: 2
strategy: RollingUpdate
image: sandeep13677/airbnb-clone:latest
resources:
  cpu: 300m mem: 256Mi

2 pods, resource limits, liveness & readiness probes.

🌐 service.yaml

kind: Service
type: NodePort
port: 80
targetPort: 3000
nodePort: 30080

Exposes app on port 30080 on the cluster node.

📈 hpa.yaml

kind: HorizontalPodAutoscaler
minReplicas: 2
maxReplicas: 5
cpu target: 70%

Auto-scales 2→5 pods when CPU exceeds 70%.

🌍 ingress.yaml

kind: Ingress
host: airbnb.domain.com
path: /
backend: airbnb-service:80

Routes domain traffic to the service (optional).

Slide 10 · Challenges & Solutions

Challenges & Solutions

⚠️ CHALLENGE 1

MongoDB Atlas Connection Refused

Docker containers had different outbound IPs — Atlas blocked them by default.

✅ SOLUTION

Added 0.0.0.0/0 to MongoDB Atlas Network Access. DB URL passed via K8s Secret.

⚠️ CHALLENGE 2

Secrets Leaking in Source Code

API keys were in .env file — dangerous to push to Git or include in Docker image.

✅ SOLUTION

Used Kubernetes Secret objects with Base64 encoding. .env deleted from image via .dockerignore.

⚠️ CHALLENGE 3

Health Probe Failing (404 on /)

Kubernetes liveness & readiness probes were hitting / which returned 404, causing pods to restart repeatedly.

✅ SOLUTION

Changed probe path to /listings in deployment.yaml. Added root redirect / → /listings in app.js.

⚠️ CHALLENGE 4

App Downtime During Updates

Restarting a single-server app causes 100% downtime for users during deployments.

✅ SOLUTION

Kubernetes Rolling Update strategy: new pods start before old ones stop. Zero downtime guaranteed.
